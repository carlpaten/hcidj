Computer Prototype

A lot has changed about our design since the beginning of the project, including hardware component choices and layout design. Since the lo-fidelity prototype, we have made the following changes:

  • FX unit layout: The usage of the effects hasn’t changed since the lo-fidelity prototype, but the FX selection has changed. We decided to align the 1,2,3 select buttons with the 3 knobs and buttons, because it is more obvious for the user to visualize right away which knob the effect is being loaded to.

  • Grouping filter with FX: Filtering is an effect itself so it makes sense for it to be with the rest of the FX. Also, in the lo-fi prototype, the Filter was next to the EQ and this was confusing.

  • EQ hardware choice: We decided the change from knob+button (like the FX) to sliders, because we did not want different functions to use the same hardware components to avoid confusion. This way, since they’re visually different, the user knows right away that the functions are different (knobs = FX and mini-sliders = EQ). Sliders communicate the nature of the equalization just as well as knobs do (middle is normal, top amplifies the frequency, and bottom attenuates it).

  • Cueing: We decided to isolate the cueing functions because in the first usability evaluation, where the cueing functions were bundled with the volume, their positioning and functionality were confusing. By isolating cueing it is clear to the user that it is a section dedicated to using their headphones. We also brought the cueing section closer to the user (it is the lowest middle section). This is intuitive because it is the only section that doesn’t affect what is being played on the speakers (close buttons are only for “me”, and all other - further - buttons are for everyone else)

Critical parts implemented at this stage:

Full layout with real hardware components: this was the most critical part for us because basically any functions can be simulated using the software, but if the right layout is not there it can not be simulated. In case a section or a function is hard or confusing to find or use, we want to know at this stage so that we can correct it for the final version. We also want to know if the way we anticipated the users to use certain components was wrong.

  • Library: this is important because it’s needed to start a set, and it is probably what will be used the most

  • Crossfader: this is important because it provides basic mixing and fading functionality, which is probably what will be used the most after the library.

  • Left Side - Play button, FX and EQ: The play button is important because it allows the user to actually play a song to be heard. The FX and EQ sections are important for now because we want to determine whether the hardware choices are right and if the user will intuitively know how to use them. We want to encourage the test subject to use these functions a lot, and having Wizard of Oz for them wouldn’t allow that to happen optimally. Only one side is needed because the other side will be the exact same.

Functionalities not implemented at this stage:

  • Jump, Loop: We judged that these functions are fairly simple to use, easy to simulate, and nothing else depends on them, so they weren’t crucial to implement. If they’re not as easy to use as we thought, the test tasks and wizard of oz are enough to determine this.

  • Rotary Encoders: the necessary hardware components have not arrived yet. This affects the Library navigation and FX select. The Library navigation will be simulated using Wizard of Oz techniques. However FX Select will not be tested, because even with proper Wizard of Oz techniques, the user will not understand that one of the 3 select buttons is always lit up (indicating which knob to load an effect to), so the related tasks would therefore probably fail.

  • Cueing: we haven’t received the external sound card yet, which is what is used to connect the headphones as a second sound output.

  • Jogging Strip: We accidentally burned the strips during the building process and were not able to get new ones in time.

  • LEDs

Usability Test

3 examiners needed: If possible, a person who has DJ or music software experience as an observer, a second observer, and a Wizard (performing Wizard of Oz techniques)

Types of users needed: a person our age (more or less 5 years) who enjoys electronic music and has never used a DJ controller before.

Usability goals:

  • The user successfully completes at least 90% of the tasks without any external help.
  • The user does not activate or deactivate any controls by accident.
  • The user does not do anything to “ruin” the sound throughout the whole test session. (The meaning of “ruining the sound” is left to the examiners to judge, and will be reported in the “Usability Test Script and Evaluation” document).
  • The controls are intuitive (the user properly uses the control in his/her first try).
  • The user finds the system easy to use.
  • The user learns how to use all the functions of the system in 30-45mins.

General result expectations:

Best-case:
- Users are able to complete all the tasks without a problem.
- By the end of the test session users are able to use the whole system without any help.
- User enjoyed using the system and would use it again.

Worst-case:
- The FX unit is confusing and user can not get the hang of how to use it.
- Users don’t use crossfader to segue between songs (they use the volume instead).
- The sound of their set doesn’t flow well (there are lots of interruptions and “bad” sounds).
- Users are overall confused with the system and don’t seem willing to continue using it or use it again.

How will the results affect the next phase?

The hardware choices we have made are final, unless something very crucial is discovered at this point, which we don’t expect will happen. What could change at this point is anything related to layout. The spacing between certain buttons might change (if some things were pressed/changed by accident, for example), or the spacing between sections (if they are too cluttered). We might change the ordering of the middle sections or the side sections, if results determine that common and liked workflows favor a different layout. For example, they might determine that the volume/crossfader should go above the library.

Heuristic Evaluation Guidelines

The following suggestions are meant to help you guide your evaluation of the HCIDJ platform. You are welcome to use them wholesale, to disregard them and create your own set of criteria, or to come up with some combination of the two.

Clarity:

  • Individual functions are easy to locate.
  • From the visual representation of a control, it is easy to figure out its associated function.
  • The first-time user is not overwhelmed with information.

Intuitiveness:

  • It is straightforward to translate thought to action.
  • The movements and actions represent well the functions they are associated to.

Flexibility:

  • The system affords the user freedom to do interesting things.
  • It is quickly decidable to the new user whether a particular action is possible or not.
  • Little frustration is experienced in the usage of the controller.

Simplicity of Design:

  • The layout and grouping of the controls helps conceptualizing their functions.
  • The user interface feels minimalist - in the sense that nothing is superfluous.

Visibility of labels and symbols:

  • The pictograms and text used are eye-catching and clearly describe the user interface.
  • The pictograms used are familiar and represent what they intuitively should.
  • There is no superfluous text; there is no missing text.

Error prevention:

  • The user interface does not suggest or emphasize courses of action that would lead to undesirable outcomes.
  • If the user accidentally messes up, he can quickly figure out how to resolve his mistake.

User Manual:

  • The user manual is short, clear and interesting; it does not discourage the first time reader.
  • The user manual is sufficient for the user to learn how to use the system without external help.
