Low-Fidelity Prototype

Main Tasks

  • Choose a song

  • Play/Pause a song

  • Change the volume of a song

  • Mix 2 songs seamlessly

  • Add Effects to a song

  • Filter out frequencies in a song

  • Hear a song in the headphones

  • Loop inside a song

  • Skip chunks of a song seamlessly

  • Jog through the song

  • Record any part of the set


Design Decisions

Main Design

We decided to group all the desired functions into independent logical groups, or units. This way the user will be able to learn The functions in each unit all have similar behaviour on the system. The units are:

  1. Volume/Cue Control Unit (x2)
  2. Library Browser Unit
  3. FX Unit (x2)
  4. Filter Unit (x2)
  5. Looping Unit (x2)
  6. Beat Jumping Unit (x2)
  7. Transport Unit (x2)
  8. Recording Unit

Fig. 1 - Main Design

Volume/Cue Control Unit

Placed in the middle because it includes the cross-fader, which is used to mix between the left and right deck. The fader being in the middle is therefore the most intuitive positioning.

Volume Control: A slider implemented using two superimposed pieces of cardboard. The smaller piece is a different colour for contrast and it is moveable up and down the bigger piece. Placed vertically to represent intensity.

Cross Fader: Same implementation as the volume control, but placed horizontally to easily visually represent the left and right deck.

Cue Buttons: Implemented using white cardboard with pictures of headphones on them so that the function is obvious.

Cue Mixer: Implemented the same way as the volume control and the cross fader. This mixes what is being heard in the headphones and what is being heard from the speakers (both can be L/R). Drawings placed so that user does not get confused with L/R mixing (cross-fader). Also, placed near the cue buttons for better visual aid.

Fig. 2 - Volume/Cue Control Unit

Library Browser Unit

The unit is placed on the right side because most people are right-handed, so they will likely use their right hand to choose a song.

Library Toggle Button: Used to toggle the music library view on the computer software. Implemented using a big green cap so that it’s very easy to see.

Folder Select Knob: Used to scroll through and select folders. Folder symbol included.

Song Select Knob: Used to scroll through songs. This is bigger than the folder knob because it will likely need to scroll through longer lists.

Left and Right Buttons: Used to load the selected song into the left or right deck.

FX Unit

FX Volume Control Knob: Used to control how much of the effect will be heard. This type of knob does not do full rotations (has a min and max) and is implemented the same way on the whole prototype controller.

FX Select Knob: Used to toggle the FX menu in the computer software, scroll through the list, and load an effect into one of the 3 positions.

Position Select Buttons (1,2,3): Used to select which position the effect will be loaded to.

Intensity Control Knobs (x3): Control the intensity of the effect.

Enable Buttons (x3): Toggle whether the effect is being applied or not. This type of button is implemented in the same way on the whole prototype controller.

High-Level Goals

The 2 essential goals and “raison d’être” of Hcidj are learnability and simplicity. These two qualities have been our motivation since the beginning and they drive us throughout the whole design process. This is why we have decided to make certain design decisions about our prototype.

For example, separating the functions into groups allows the user to first identify to a group or section when learning a task, and then learning the task itself. Having a few groups instead of a lot of buttons to choose from, one can learn more functions faster.

We kept the design simple by omitting some functionalities that we thought are too advanced for beginners and unnecessary to include, such as tempo control. We also integrated other functionalities into our system without putting controls for them on the controller, such as automatic track synchronization.
